> **対象読者**: これから R2C AIチャットを自社サイトに導入するパートナー事業者様、およびパートナーに寄り添って立ち上げを進める R2C 側の担当者。
> **前提**: ITに詳しくない方でも、このドキュメントを上から順に読めば、問い合わせの受付からチャット導入 → 本番運用までを迷わず進められる構成になっています。

## 1. Executive Summary（全体像の要約）

R2C AIチャットをパートナーサイトに導入する流れは、**3つのフェーズ**に分かれます。

| フェーズ | 期間の目安 | 主な作業 | 完了の合図 |
|---|---|---|---|
| **Phase1: ヒアリング** | 1〜3日 | 業種・主要コンバージョン・IT環境の聞き取り | ヒアリングシート記入完了 |
| **Phase2: ドライラン** | 1〜2週間 | ステージングでの動作確認とコンバージョン計測テスト | Admin UI にテスト CV が表示される |
| **Phase3: 本番ロールアウト** | 1日 | 本番サイトへのウィジェット配置、監視開始 | 実ユーザーの CV が 1 件以上記録される |

**パートナー側でやっていただくこと（最小）**:

1. サイトのドメイン・完了ページ URL をお知らせいただく
2. R2C から発行する JavaScript 1行をヘッダーかフッターに貼る
3. 購入・問い合わせ・予約などの「完了ページ」に計測コードを1ブロック貼る

**R2C 側でやること**:

- テナント（お客様専用の領域）作成と API キー発行
- 対象ドメインを CORS 許可リストに追加
- ウィジェット埋め込みコードの生成
- Admin UI の使い方レクチャー（1回 30分程度）

## 2. このプレイブックが必要な理由

これまでの導入では、以下の問題が繰り返し発生していました。

- **「どのイベントをコンバージョンとするか」で迷走する**: 業種ごとに正解が違うのに、毎回ゼロから議論していた
- **サイト側の JavaScript 埋め込みで失敗する**: 非同期読み込みのタイミングでエラーが出ていた
- **Admin UI の使い方がわからず放置される**: 運用開始後にログインしないまま数週間経ってしまうケース
- **ロールバック手順が共有されていない**: 「とりあえず動いているから触るな」状態に陥る

本書は、この4つをすべて解消することを目的としています。

### 2.5 前提条件チェックリスト

導入作業を開始する前に、以下の項目をすべて確認してください。**1つでも未確認の項目があると、途中で手戻りが発生します。**

#### 2.5.1 ドメイン・CORS 許可 Origin 確認

R2C のサーバーは、許可したドメインからしかウィジェットの通信を受け付けません（セキュリティのため）。

- [ ] パートナーサイトの本番ドメイン（例: `https://shop.example.com`）
- [ ] ステージング環境のドメイン（例: `https://staging.shop.example.com`）
- [ ] サブドメインを複数使う場合は、それぞれのドメイン
- [ ] `www` あり / なしの両方で配信している場合は両方

上記を R2C 担当者にお伝えください。R2C 側で `ALLOWED_ORIGINS` 環境変数に追加します。

**確認コマンド（R2C 担当者用）**:
```bash
# VPS 側で設定を確認
ssh root@65.108.159.161 'grep ALLOWED_ORIGINS /opt/rajiuce/.env'
```

#### 2.5.2 API キー払い出しフロー

API キーはウィジェットが R2C サーバーと通信するための鍵です。**サイトに埋め込むコードに含まれるため、一般ユーザーにも見えます**（これは正常な挙動で、サーバー側でドメイン検証を行っています）。

1. R2C 担当者がテナントを作成（Admin UI `/admin/tenants` もしくは SQL）
2. API キーは `rjc_` で始まる 72 文字程度の文字列として発行される
3. パートナーへ **専用フォルダ経由で共有**（メール平文送信は避ける）
4. パートナーは埋め込みコードに記載して使用

**キーの取り扱いルール**:
- パブリックなソースコードリポジトリにコミットしない
- キーが漏洩した疑いがあれば、R2C 担当者に連絡してローテーション

#### 2.5.3 パートナー側 IT 環境ヒアリング項目

以下をパートナーに確認してください。回答は Phase1 のヒアリングシートに転記します。

- [ ] サイトのホスティング（WordPress / Shopify / 自社サーバー / その他）
- [ ] 現在サイトに入っている計測タグ（Google Analytics / Google Tag Manager / Facebook Pixel 等）
- [ ] サイトの HTML を編集できる担当者の有無（社内エンジニア / 外部制作会社）
- [ ] 「完了ページ」に該当する URL のリスト
- [ ] サイトに SPA（React / Vue / Next.js 等）が使われているか
- [ ] Cookie 同意バナー・GDPR 対応の有無

**SPA の場合の注意**: 完了ページが URL 遷移ではなくコンポーネント表示の場合、CV 発火タイミングを JavaScript で制御する必要があります（第5章 パターン C を参照）。

## 3. R2C テナントオンボーディングフロー

パートナーがお客様として登録されるまでの作業手順です。

### 3.1 テナント登録

1. Admin UI にログイン（super_admin 権限が必要）
2. `/admin/tenants` を開き、「新規テナント作成」ボタンをクリック
3. 以下を入力:
   - **テナント ID**: 英小文字・数字・ハイフンのみ（例: `bross-niigata`）。後から変更不可なので要確認
   - **表示名**: Admin UI 上で表示される名前（例: `BROSS新潟`）
   - **features**: `rag: true` を既定、その他はパートナーの契約プランに応じて設定
4. 「保存」すると API キーが自動発行されます

### 3.2 ウィジェット埋め込みコードの生成

テナント作成後に発行された API キーを使い、以下のスニペットを生成してパートナーに渡します。

```html
<!-- R2C AI チャットウィジェット -->
<script
  src="https://api.r2c.biz/widget.js"
  data-tenant="<テナントID>"
  data-api-key="<発行されたAPIキー>"
  async
></script>
```

**貼り付け位置の推奨**: `</body>` 直前、または Google Tag Manager 等のタグマネージャー経由。

### 3.3 初期テスト

以下の3点を確認したら Phase1 完了です。

- [ ] パートナーサイトを開いて、右下にチャットアイコンが表示される
- [ ] アイコンをクリックして、チャットパネルが開く
- [ ] 任意のメッセージを送信して、AI からの返答がある

問題があれば第11章「トラブルシューティング FAQ」を参照。

## 4. 主要コンバージョンイベントカタログ

R2C のコンバージョン種別（`conversion_type`）は以下の5種類です。業種によって「何をコンバージョンとするか」が異なるため、パートナーと事前にすり合わせてください。

### 4.1 種別一覧

| 種別 | 意味 | 金額(value) の扱い |
|---|---|---|
| `purchase` | 商品の購入・決済完了 | 購入金額（円） |
| `inquiry` | 問い合わせフォーム送信 | 省略可（0 でも可） |
| `reservation` | 予約・来店予約の完了 | 省略可（見込み単価を入れてもよい） |
| `signup` | 会員登録・資料請求 | 省略可 |
| `other` | 上記に該当しないアクション | 任意 |

上記以外の値は **400 invalid_request エラー**になります。必ずこのどれかを選んでください。

### 4.2 業種別の該当 CV 例

**中古車販売（例: BROSS新潟）**

| サイトイベント | `conversion_type` | value の例 |
|---|---|---|
| 車両購入申し込みフォーム送信 | `purchase` | 車両価格（例: 2,890,000） |
| 試乗予約フォーム送信 | `reservation` | 省略（0） |
| 問い合わせフォーム送信 | `inquiry` | 省略（0） |

**不動産仲介**

| サイトイベント | `conversion_type` | value の例 |
|---|---|---|
| 資料請求送信 | `signup` | 省略 |
| 内見予約送信 | `reservation` | 想定家賃 × 12（年換算の期待 LTV） |
| 問い合わせ送信 | `inquiry` | 省略 |

**SaaS（月額サブスク）**

| サイトイベント | `conversion_type` | value の例 |
|---|---|---|
| 無料トライアル申し込み | `signup` | 省略 |
| 有料プラン契約完了 | `purchase` | 初月金額または年額 |
| セールス問い合わせ | `inquiry` | 省略 |

**教材販売（オンライン講座）**

| サイトイベント | `conversion_type` | value の例 |
|---|---|---|
| 決済完了 | `purchase` | 販売価格 |
| 無料サンプル請求 | `signup` | 省略 |
| LINE 登録 | `other` | 省略（要協議） |

### 4.3 パートナーと決めるべきこと

- **「主要 CV」は 1〜3 種類に絞る**: 全部計測しようとすると分析時に埋もれます
- **value の単位をそろえる**: 円ならすべて円、ドルならすべてドル
- **テストデータを本番と分ける**: ステージング用には別テナントを発行するか、`conversion_type: 'other'` でタグ付けする運用も検討

## 5. trackConversion() 埋め込み標準スニペット集

> **重要**: R2C には **2つの CV 送信経路** があります。用途に応じて使い分けてください。

### 5.1 2経路の違い

| 経路 | 用途 | 認証 | 送信先 | 呼び出し元 |
|---|---|---|---|---|
| **Widget 経路** | パートナーサイトのブラウザ上から送信 | `x-api-key`（ウィジェットが自動付与） | `POST /api/events`（`event_type: 'chat_conversion'`） | `window.r2c.trackConversion()` または `window.r2cQueue` |
| **サーバー間経路** | パートナー側サーバー → R2C サーバー | `x-api-key`（パートナーサーバーが明示付与） | `POST /api/conversion/attribute` | パートナー側バックエンドの HTTP リクエスト |

**どちらを選ぶか**:
- **ブラウザから完結**（ほぼ全ケース） → Widget 経路
- **決済サーバーやバックエンド処理で確定する場合** → サーバー間経路（改ざん防止・確実性が高い）
- **両方使ってもよい**が、同一 CV を二重計上しないよう注意

### 5.2 Widget 経路（推奨）

Widget 経路は、パートナーが自分でサーバー実装を書かなくてよいため、ほぼすべてのケースでこちらを推奨します。

#### パターン A: キュー方式（最も安全）

ウィジェットの読み込みより**前**にキューへ積むだけ。非同期読み込みのタイミングに依存しません。

```html
<script>
  window.r2cQueue = window.r2cQueue || [];
  window.r2cQueue.push({
    type: 'conversion',
    conversionType: 'purchase',
    value: 50000
  });
</script>
<script
  src="https://api.r2c.biz/widget.js"
  data-api-key="<APIキー>"
  async
></script>
```

#### パターン B: 直接呼び出し

ボタンクリック等、ウィジェットがすでにロード済みの状態で呼び出せます。

```html
<script>
  document.getElementById('purchase-btn').addEventListener('click', function() {
    window.r2c.trackConversion('purchase', 29800);
  });
</script>
```

#### パターン C: SPA（React / Vue / Next.js）

ページ遷移ではなくコンポーネント表示で完了する場合は、マウント時に呼び出してください。

```javascript
// 購入完了画面コンポーネント
useEffect(() => {
  if (window.r2c && window.r2c.trackConversion) {
    window.r2c.trackConversion('purchase', orderTotal);
  } else {
    // フォールバック: キュー経由
    window.r2cQueue = window.r2cQueue || [];
    window.r2cQueue.push({
      type: 'conversion',
      conversionType: 'purchase',
      value: orderTotal,
    });
  }
}, []); // マウント時に1度だけ
```

#### パターン D: 問い合わせフォーム送信後

```html
<script>
  document.getElementById('inquiry-form').addEventListener('submit', function() {
    window.r2cQueue = window.r2cQueue || [];
    window.r2cQueue.push({
      type: 'conversion',
      conversionType: 'inquiry',
    });
  });
</script>
```

#### パターン E: 電話タップ（モバイル特化）

```html
<a href="tel:0120-xxx-xxx" id="phone-link">お電話はこちら</a>
<script>
  document.getElementById('phone-link').addEventListener('click', function() {
    window.r2cQueue = window.r2cQueue || [];
    window.r2cQueue.push({
      type: 'conversion',
      conversionType: 'inquiry',
    });
  });
</script>
```

### 5.3 サーバー間経路

パートナー側のバックエンドから直接 R2C に送信する場合に使います。決済完了 Webhook からの送信など、ブラウザの改ざんリスクを避けたい場面で有効です。

**エンドポイント**: `POST https://api.r2c.biz/api/conversion/attribute`
**認証**: `x-api-key: <APIキー>`
**Content-Type**: `application/json`

**リクエスト例**:

```bash
curl -X POST https://api.r2c.biz/api/conversion/attribute \
  -H "Content-Type: application/json" \
  -H "x-api-key: <APIキー>" \
  -d '{
    "session_id": "<UUID形式のチャットセッションID>",
    "conversion_type": "purchase",
    "conversion_value": 29800,
    "trigger_type": "checkout_complete",
    "message_count": 12
  }'
```

**レスポンス**:
- 成功: `202 { "accepted": true }`
- バリデーションエラー: `400 { "error": "invalid_request", "details": [...] }`
- 認証エラー: `401 { "error": "tenant_not_found" }`

**必須フィールドは `conversion_type` のみ**です。その他はすべて省略可（`session_id` / `visitor_id` を省略するとチャットとの紐づけ精度は下がります）。

### 5.4 二重計上を避けるコツ

```javascript
// 同一セッション内で1度だけ送信
if (!sessionStorage.getItem('cv_sent')) {
  window.r2cQueue = window.r2cQueue || [];
  window.r2cQueue.push({ type: 'conversion', conversionType: 'purchase', value: 50000 });
  sessionStorage.setItem('cv_sent', '1');
}
```

## 6. Phase1: パートナーヒアリング

Phase1 のゴールは、**パートナー業種・主要 CV・サイト構造・IT 環境**を1枚のシートに落とし込むことです。

### 6.1 ヒアリングシートの入手

既存の[`docs/PARTNER_PSYCHOLOGY_INTERVIEW.md`](./PARTNER_PSYCHOLOGY_INTERVIEW.md) をベースに、パートナーの業種に合わせて書き換えて使用します。

**注意**: 元のシートは英語教材販売の文脈で書かれているため、以下の項目は業種汎用に置き換えてください。

### 6.2 汎用業種向けに書き換える設問例

| 元の設問（英語教材特化） | 汎用版（業種非依存） |
|---|---|
| Q1「商材の名前（または通称）」 | 「主力商品・サービス名」 |
| Q2「価格はいくらでしたか？」 | 「主力商品・サービスの価格帯（単価 or 月額）」 |
| Q6「どんな年齢層・職業の方が購入？」 | 「想定顧客の年齢層・業種・役職」 |
| Q7「典型的なお客さんはどんな悩み？」 | 「顧客の主要な悩み・ペインポイント 上位3つ」 |
| Q13「主にどのチャネルでセールス？」 | 「主要な集客・販売チャネル」 |
| Q19「よくあった断り文句」 | 「よくある購入検討停止の理由 上位3つ」 |
| Q25「顧客リストは今も手元にありますか？」 | 「既存顧客リストの有無・件数」 |

### 6.3 必ず確認する 3 点

業種を問わず、以下の3点は必ず聞き出してください。これが決まらないと Phase2 に進めません。

1. **主要コンバージョン**: 何を「成果」と定義するか（第4章の `conversion_type` のどれか）
2. **運営メンバーの IT リテラシ**: Admin UI を操作できる人は誰か、何人か
3. **サイト構造**: 完了ページの URL、SPA かどうか、既存タグマネージャーの有無（2.5.3 を再確認）

### 6.4 心理・セールス深掘り（任意）

パートナーが過去に高単価商材を販売した経験をお持ちの場合、[`docs/PARTNER_PSYCHOLOGY_INTERVIEW.md`](./PARTNER_PSYCHOLOGY_INTERVIEW.md) の セクション4（心理学的トーク）・セクション5（実績データ）は、AI の会話トーン調整やチューニングルール設計に使えます。これは必須ではなく「活用できるなら活用」のオプション項目です。

## 7. Phase2: ドライラン（1〜2週間）

本番公開前に、ステージング環境で実際の導入コードが動くか確認します。

### 7.1 ドライランのゴール

- チャットが表示され、AI 応答が返ること
- 各 `conversion_type` のテストイベントが Admin UI に記録されること
- パートナー側の既存分析ツール（GA4 等）との件数乖離が把握できること

### 7.2 ステージング環境でのウィジェット配置

本番と同じコードをステージングサイトに貼り付けます。

```html
<script
  src="https://api.r2c.biz/widget.js"
  data-tenant="<テナントID>"
  data-api-key="<APIキー>"
  async
></script>
```

**注意**: ステージング用ドメインも R2C 側の CORS 許可リストに入っている必要があります（2.5.1 を参照）。

### 7.3 E2E 検証手順

以下をパートナーと一緒に（または画面共有で）実施します。

1. ステージングサイトを開く
2. チャットアイコンが右下に表示されることを確認
3. チャットで任意のメッセージを送信、AI 応答を受信
4. 「完了ページ」に遷移する一連のフロー（フォーム入力 → 送信 → サンクスページ表示）を実施
5. サンクスページで `trackConversion()` が呼ばれることをブラウザの DevTools → Network タブで確認
   - `POST /api/events` リクエストが 200 OK で返ってくること
   - リクエストボディに `event_type: 'chat_conversion'` と正しい `conversion_type` が含まれていること

### 7.4 Admin UI でのコンバージョン確認

ドライラン開始から 1〜2分以内に、以下のパスで CV が記録されていることを確認します。

- **パス**: `https://admin.r2c.biz/admin/conversion`
- **確認項目**:
  - [ ] 「総コンバージョン数」が増えている
  - [ ] 「コンバージョン種別内訳」に期待した種別が表示される
  - [ ] クリックで個別 CV の詳細（session_id / 送信時刻）が見られる

### 7.5 件数整合性の取り方

パートナー側の既存計測ツール（GA4 等）とのズレを把握します。

| 比較対象 | R2C 側の集計場所 | パートナー側の集計場所 |
|---|---|---|
| 総 CV 件数 | `/admin/conversion` 画面の「総コンバージョン数」 | GA4 の「コンバージョン」レポート |
| 種別ごとの件数 | 同画面の「種別内訳」 | GA4 のイベント別レポート |

**乖離が生じる主な理由**:
- R2C は `x-api-key` での認証失敗時は計測できない（GA4 は常に集計）
- 広告ブロッカーが `api.r2c.biz` をブロックするケース
- SPA で URL 遷移時に両方から呼んでいる場合のカウント差

**目安**: 乖離率 20% 以内なら正常範囲、50% を超える場合は CV 発火位置の見直しが必要です。

## 8. Phase3: 本番ロールアウト

ドライランで問題がなければ、本番サイトへ移行します。

### 8.1 ロールアウト手順

1. パートナーサイトの本番ドメインを R2C の CORS 許可リストに追加（未追加の場合）
2. 本番サイトにウィジェット埋め込みコードを配置（ヘッダーまたはフッター）
3. 本番用サンクスページに `trackConversion()` を配置
4. デプロイ後、実際のユーザーの CV が `/admin/conversion` で確認できるまで待機（初回 CV はアクセス数次第）
5. 初回 CV 確認後、パートナーに「運用フェーズ開始」を通知

### 8.2 公開直後の監視（24時間）

- [ ] `POST /api/events` の 4xx / 5xx エラー率をログで監視
- [ ] `https://admin.r2c.biz/admin/analytics` でチャット利用者数の推移を確認
- [ ] パートナーからの「動かない」連絡がないか 1 時間おきに確認

### 8.3 運用定例化（推奨）

- **週次**: Admin UI の「会話分析ダッシュボード」をパートナー含めて確認
- **月次**: チューニングルール見直し、CV 率の傾向レビュー
- **随時**: 第11章のトラブルシューティング FAQ を参照

### 8.5 Admin UI 操作ガイド（パートナー向け）

> **対象**: パートナー側の運用担当者（IT に詳しくなくても大丈夫です）
> **所要時間**: 初回レクチャー 30分程度

#### 8.5.1 管理画面ログイン手順

1. ブラウザで `https://admin.r2c.biz` を開く
2. R2C から事前に伝えた**メールアドレス**と**パスワード**を入力
3. 「ログイン」ボタンをクリック
4. ダッシュボード画面が表示されれば成功

**ログインできないとき**:
- パスワードを忘れた場合は R2C 担当者にリセット依頼
- 3回ほど失敗すると一時ロックされる場合があります（少し時間を置いてから再試行）

#### 8.5.2 コンバージョン（CV）の確認

1. 画面上部のナビから「**成果管理**」→「**コンバージョン分析**」をクリック
   （URL: `/admin/conversion`）
2. 右上の期間セレクターで「7日 / 30日 / 90日」を切り替え
3. 画面中央に表示される項目を確認
   - 総コンバージョン数
   - 種別ごとの内訳（purchase / inquiry / reservation / signup / other）
   - 平均温度スコア（お客様の関心度合いを 0〜100 で数値化）
   - トップ心理アプローチ（効果的だった会話スタイル）

#### 8.5.3 チャット履歴を見る

1. ナビから「**会話**」→「**会話履歴**」をクリック
   （URL: `/admin/chat-history`）
2. 日付やキーワードで絞り込み
3. 気になる会話の行をクリックで、やり取りの全文が見られます

**プライバシー配慮**:
- 個人情報（名前・電話番号など）が会話に含まれる場合があります
- 社内での閲覧・共有範囲は運用ルールに従ってください

#### 8.5.4 温度感スコア・声がけ設定

「お客様が興味を示しているか」をスコア化し、一定以上になったらチャットから自動で声をかける機能です。

1. ナビから「**声がけ**」→「**声がけ設定**」をクリック
   （URL: `/admin/engagement`）
2. 「発動しきい値（温度スコア）」「声がけ文言」を調整
3. 保存後、新規セッションから設定が反映されます

**最初は既定値のまま運用し、1〜2ヶ月経過してから調整することを推奨します。**

#### 8.5.5 低 IT リテラシ向けステップバイステップ

**毎朝のルーティン例（5分）**:

1. `https://admin.r2c.biz` を開いてログイン
2. トップ画面の「CV計測: ON」バッジが緑色になっていることを確認
3. ナビの「成果管理」→「コンバージョン分析」で昨日の件数を見る
4. ナビの「会話」→「会話履歴」で特に気になった会話を 2〜3 件開く
5. 異変（エラーメッセージ / 件数が極端に少ない等）があれば R2C 担当者に連絡

**「触ってはいけない」箇所**:
- 「テナント管理」メニュー（super_admin 専用・通常パートナーには表示されません）
- API キーの再発行（影響範囲が大きいため、必ず R2C 担当者と相談）

## 9. ロールバック手順

チャットを一時的に停止したい、もしくは完全に外したい場合の手順です。

> **重要な前提**: 現時点で R2C には「コンバージョン計測のみを無効化する」専用フラグ（`features.conversion_tracking`）は**まだ実装されていません**。代替として以下の 2 通りで対応します。

### 9.1 パターン A: イベント計測全体を一時停止

Admin UI の操作ではなく、SQL もしくは API で `features.event_tracking` を `false` にします。

```sql
-- 該当テナントのイベント計測を無効化
UPDATE tenants
SET features = jsonb_set(features, '{event_tracking}', 'false')
WHERE id = '<テナントID>';
```

- 実行後、ウィジェットは引き続き表示されますが、CV を含む各種イベント送信が止まります
- 再有効化は `'false'` を `'true'` に置き換えるだけ

### 9.2 パターン B: ウィジェット自体を外す（完全停止）

パートナーサイトの HTML から以下のスクリプトタグを削除してください。

```html
<script
  src="https://api.r2c.biz/widget.js"
  data-api-key="..."
  async
></script>
```

- チャット UI・CV 計測・会話ログ すべての送信が停止します
- 次回デプロイまたはページ更新で反映されます

### 9.3 既存の features フラグ一覧

`tenants.features` JSONB カラムで管理されるフラグです。

| キー | 意味 | 既定値 | 実装状況 |
|---|---|---|---|
| `rag` | RAG（知識検索）を有効化 | `true` | ✅ 実装済み |
| `voice` | 音声応答（LiveKit + TTS） | `false` | ✅ 実装済み |
| `avatar` | アバター表示 | `false` | ✅ 実装済み |
| `deep_research` | 深掘り検索機能 | `false` | ✅ 実装済み |
| `event_tracking` | イベント送信の有効化 | `false`（明示有効化が必要） | ✅ 実装済み |
| `conversion_tracking` | CV 計測のみを制御 | N/A | ⚠️ **未実装（将来 TODO）** |

**`conversion_tracking` の実装予定**:
現時点では `event_tracking` が OFF のときに CV も一緒に止まります。CV のみを選択的に無効化するニーズが複数パートナーから発生した段階で、専用フラグを追加する方針です。

## 10. KPI

ロールアウト後に継続的に観測する指標です。

### 10.1 KPI 一覧

| 指標 | 計測方法 | 目安 |
|---|---|---|
| **CV 発火率** | `GET /v1/admin/conversion/attributions` で期間内件数を取得 | 業種により異なる |

### 10.2 KPI の取得クエリ例

```bash
# 過去 30 日の CV 件数（種別内訳付き）
curl -H "Authorization: Bearer <JWT>" \
  "https://api.r2c.biz/v1/admin/conversion/attributions?period=30d"
```

レスポンスに含まれる `byType` がそのまま種別内訳として使えます。

### 10.5 費用・プラン説明テンプレート（低優先）

> **優先度**: 低（本書の主題ではありませんが、パートナー商談時によく聞かれる項目のためテンプレを用意）

#### 10.5.1 Stripe プラン説明雛形

```
R2C AIチャットのご利用料金は、以下の 2 つの合算となります。

1. 固定月額（プラットフォーム利用料）
   - ベーシック: 月額 xx,xxx 円
   - スタンダード: 月額 xx,xxx 円
   - エンタープライズ: 個別見積もり

2. 従量課金（AI 呼び出し回数 × 単価）
   - お支払いは Stripe 経由で月次請求

詳細な価格は契約時にご提示します。
```

#### 10.5.2 月次概算 $27-48 の根拠

これは **R2C 側のインフラ運用コスト**の目安です（パートナー料金とは別）。

| 項目 | 月額 |
|---|---|
| VPS (Hetzner) | 約 $20 |
| Groq API 使用料 | 約 $5〜20（利用量依存、120B 比率 ≤ 10%） |

一部パートナーでは「成果報酬型」での契約を打診される場合があります。

- CV 1件あたりの単価を固定
- 月次の CV 集計は `/admin/conversion` のデータを基準とする
- 最低保証の有無は個別協議

**注意**: 現時点で Stripe 請求基盤は固定 + 従量の二層構造のため、純粋な成果報酬型はカスタム運用となります。まずは固定プランからスタートし、3 ヶ月後に再契約で成果報酬型へ移行するパターンが現実的です。

## 11. トラブルシューティング FAQ

### Q1. ウィジェットが表示されない

**チェック項目**:
- [ ] ブラウザの DevTools → Console にエラーが出ていないか
- [ ] `https://api.r2c.biz/widget.js` へのリクエストが 200 OK で返っているか（Network タブ）
- [ ] `data-api-key` の値に余分なスペースや改行が入っていないか
- [ ] パートナーのドメインが CORS 許可リストに入っているか（2.5.1）

### Q2. CV が発火しない

**チェック項目**:
- [ ] 完了ページにもウィジェット（`widget.js`）が読み込まれているか
- [ ] `conversion_type` が `purchase` / `inquiry` / `reservation` / `signup` / `other` のどれかになっているか（それ以外は 400 エラー）
- [ ] DevTools → Network で `POST /api/events` が送信されているか
- [ ] レスポンスが 200 OK か（401 ならキー、400 ならリクエスト形式、5xx ならサーバー側）
- [ ] キュー方式（パターン A）で、ウィジェットより前に `window.r2cQueue.push()` を呼んでいるか

### Q3. CORS エラーが出る（`Access-Control-Allow-Origin`）

ブラウザ Console に以下のようなエラーが出る場合：

```
Access to fetch at 'https://api.r2c.biz/api/events' from origin
'https://shop.example.com' has been blocked by CORS policy
```

- R2C 担当者に連絡し、`ALLOWED_ORIGINS` に該当ドメインを追加してもらってください
- 追加後は R2C 側で API プロセスを再起動する必要があります（数分でオンライン）

### Q4. Admin UI にログインできない

- パスワードリセットは R2C 担当者に依頼
- メールアドレスの大文字小文字を再確認
- Cookie をブロックするブラウザ設定の場合は許可してから再試行

### Q5. CV 件数がパートナー側ツールと大きく違う

- 乖離率 20% 以内なら正常範囲（7.5 参照）
- 広告ブロッカー・プライバシーブラウザが影響している可能性
- SPA で両方から二重送信していないか確認

### Q6. チャットの AI 応答が期待と違う

- Admin UI の「ナレッジ管理」（`/admin/knowledge`）で FAQ を追加
- 「チューニングルール」（`/admin/tuning`）でトーンやテンプレを調整
- 変更は数分〜数時間で反映

### Q7. 緊急で計測・表示を止めたい

- 一時停止 → 第9章パターン A（SQL で `event_tracking` を false）
- 完全停止 → 第9章パターン B（HTML からスクリプトタグ削除）

### Q8. API キーが漏れた可能性がある

- 即座に R2C 担当者に連絡
- 担当者が新しい API キーを発行、パートナーは HTML の `data-api-key` を更新
- 旧キーは無効化されます
